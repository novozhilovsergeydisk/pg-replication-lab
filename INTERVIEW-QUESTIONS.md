# Вопросы на собеседовании DBA: физическая репликация

## 20 вопросов по физическому резервированию

### 1. Чем физическая репликация отличается от логической?

Физическая копирует весь кластер на уровне WAL (все базы, схемы, роли).
Логическая — через декодирование WAL, можно реплицировать отдельные таблицы,
между разными версиями PostgreSQL, на разные схемы.

### 2. Что такое WAL и зачем он нужен?

Write-Ahead Log — журнал предзаписи. PostgreSQL сначала пишет изменения в WAL,
потом в data-файлы. Нужен для: восстановления после сбоя (redo), репликации,
point-in-time recovery (PITR).

### 3. Какие бывают уровни WAL?

`minimal` — только для восстановления после сбоя (нельзя PITR и репликацию)
`replica` — PITR + физическая репликация (стандарт с PostgreSQL 10)
`logical` — всё выше + логическая репликация

### 4. Что такое slot репликации?

Физический слот гарантирует, что primary не удалит WAL, который нужен
отставшему или отключившемуся standby. Без слота при отставании WAL может
быть перезаписан, и реплику придётся пересоздавать через pg_basebackup.

### 5. Чем отличается синхронная репликация от асинхронной?

- **Синхронная:** primary ждёт подтверждения от standby перед коммитом.
  Гарантия 0 потерь. Выше latency на запись.
- **Асинхронная:** primary не ждёт. Быстрее, но возможна потеря последних
  транзакций при внезапном падении primary.

### 6. Какая репликация стоит по умолчанию?

Асинхронная. Синхронную нужно явно включать через `synchronous_standby_names`.

### 7. Что такое pg_rewind и в каких случаях он нужен?

Утилита для «отката» старого primary после promote одного из standby.
Вместо полного pg_basebackup — синхронизирует только разошедшуюся часть WAL.
Работает только при `wal_log_hints = on` или `data_checksums = on`.

### 8. Что будет, если standby отстал и WAL уже перезаписан?

Репликация сломается. Придётся заново снимать pg_basebackup. Слот
репликации предотвращает эту ситуацию.

### 9. Что такое timeline в PostgreSQL?

Счётчик переключений primary. Каждый promote реплики создаёт новый timeline.
Нужен, чтобы pg_rewind и WAL-архивация понимали, на какой ветке истории
находится сервер.

### 10. Как проверить отставание реплики?

```sql
-- на primary:
SELECT application_name, write_lag, flush_lag, replay_lag FROM pg_stat_replication;

-- на standby:
SELECT pg_wal_lsn_diff(pg_last_wal_receive_lsn(), pg_last_wal_replay_lsn());
```

### 11. Почему может не работать репликация? Топ-5 причин

1. Firewall блокирует порт
2. Нет записи в pg_hba.conf для standby
3. Не совпадает пароль репликации
4. Разные мажорные версии PostgreSQL
5. Нет места на диске standby

### 12. Что делает pg_stat_replication?

Системное представление primary, показывающее все подключённые standby:
адрес, состояние потока, отставание, имя приложения.

### 13. Можно ли читать данные со standby?

Да, если `hot_standby = on` (по умолчанию). Без этого параметра standby
не отвечает на запросы — только применяет WAL.

### 14. Что такое cascade replication?

Standby может принимать репликацию не от primary, а от другого standby.
Снижает нагрузку на primary. Настраивается через `primary_conninfo` с
указанием промежуточного standby.

### 15. Как переключить primary без потери данных (switchover)?

Перевести синхронную реплику в sync-режим, дождаться `sync_state = sync`,
выполнить promote на ней, переключить приложения. В асинхронном режиме —
swap ролей всегда с риском потери последних транзакций.

### 16. Что будет с репликацией, если primary перезагрузить?

Standby обнаружит разрыв соединения, будет пытаться переподключиться
периодически. После старта primary — переподключится и догонит изменения.
При использовании слота — WAL будет сохранён, отставание нагонится.

### 17. Что такое pg_basebackup?

Утилита для создания базовой копии кластера «на лету», без остановки БД.
Используется для первоначальной настройки standby.

### 18. Какой минимальный набор параметров нужен для репликации на primary?

```ini
wal_level = replica
max_wal_senders = 2-10   # минимум 1 на standby, лучше с запасом
```

И строка в pg_hba.conf:
```
host replication user ip/32 md5
```

### 19. Можно ли настроить репликацию если primary уже работает?

Да. Все параметры (wal_level, max_wal_senders) меняются на лету через
`pg_reload_conf()` или рестарт, pg_hba.conf тоже reload. pg_basebackup
снимает копию без остановки.

### 20. Чем потоковая репликация отличается от файловой?

- **Потоковая (streaming):** WAL передаётся по TCP сразу, без задержки.
  Современный стандарт.
- **Файловая (archive):** standby забирает WAL из archive-директории
  (например, через `restore_command`). Задержка больше, но не требует
  прямого сетевого соединения. Используется редко, обычно как fallback.

---

### 21. Как настроить репликацию с нуля, если primary уже работает?

Изменить `wal_level = replica`, `max_wal_senders = 5` (требует рестарта),
добавить строку в `pg_hba.conf`, перезагрузить, создать пользователя
`replicator`, снять `pg_basebackup` на standby, создать `standby.signal`,
указать `primary_conninfo`, запустить standby.

### 22. Что такое timeline switch и когда он происходит?

При `pg_ctlcluster promote` создаётся новый timeline. Нужен, чтобы отличать
WAL, записанный до promote, от WAL после promote. Без `pg_rewind` старый
primary не сможет подключиться к новому — у него «прошлый» timeline.

### 23. Можно ли подключиться к standby для записи?

Нет. Физический standby всегда read-only. Попытка INSERT/UPDATE/DELETE
вернёт ошибку: `cannot execute INSERT in a read-only transaction`.
Исключение — logical replication, но это другая технология.

### 24. Как проверить, что слот репликации активен?

```sql
SELECT slot_name, slot_type, active, restart_lsn, xmin FROM pg_replication_slots;
```

- `active = t` — слот используется подключённым standby
- `active = f` — standby отключился, WAL не удаляется (следить!)
- `restart_lsn` — позиция WAL, которую standby уже подтвердил

### 25. Что делать, если слот репликации разросся?

Если standby отключён надолго — слот удерживает WAL. Диск на primary может
заполниться. Варианты:
- Подключить standby и дать догнаться
- Удалить слот (`pg_drop_replication_slot`) и пересоздать standby через
  pg_basebackup (потеряете данные, если standby нужен для failover)
- Ограничить `max_slot_wal_keep_size` (PG 13+)

## 10 других вопросов по PostgreSQL в целом

### 1. Что такое autovacuum и почему он важен?

Автоматический сборщик мусора. Удаляет «мёртвые» версии строк (MVCC).
Без него — разрастание таблиц, падение производительности, сбои из-за
переполнения счётчика транзакций (wraparound). Отключать autovacuum —
грубая ошибка в production.

### 2. Что такое MVCC?

Multi-Version Concurrency Control. Каждое изменение создаёт новую версию
строки, старая остаётся. Читающие запросы не блокируют пишущие, и наоборот.
Это позволяет параллельный доступ без блокировок.

### 3. Какие есть типы индексов в PostgreSQL?

- B-tree — по умолчанию, почти всё
- Hash — точное равенство
- GiST — поиск по геоданным, полнотекстовый
- GIN — массивы, jsonb, полнотекстовый
- BRIN — очень большие таблицы, где данные физически упорядочены
- SP-GiST — пространственные данные
- Bloom — фильтр Блума

### 4. Что такое EXPLAIN ANALYZE?

План выполнения запроса с реальным временем выполнения. Показывает:
как PostgreSQL читает данные, какие индексы использует, где узкие места.
Ключевой инструмент оптимизации запросов.

### 5. Какие есть способы резервного копирования PostgreSQL?

- pg_dump — логический дамп (одна БД, SQL)
- pg_dumpall — все БД и глобальные объекты
- pg_basebackup — физическая копия кластера
- WAL-архивация — непрерывное архивирование WAL (PITR)
- snapshots — снапшоты на уровне файловой системы (ZFS, LVM)

### 6. Что такое pg_hba.conf и как он работает?

Файл аутентификации клиентов. PostgreSQL проверяет каждое подключение
сверху вниз по правилам: тип подключения (local/host/hostssl), база,
пользователь, IP-адрес, метод (peer/scram-sha-256/trust/cert). Первое
совпавшее правило применяется.

### 7. Какие есть режимы работы pg_dump?

- `pg_dump` — дамп одной БД в SQL или custom-формат (сжатый, можно
  восстанавливать выборочно через pg_restore)
- `pg_dump -Fc` — custom-формат, самый гибкий
- `pg_dumpall` — глобальный дамп всех БД + роли + табличные пространства
- `pg_dump -j N` — параллельный дамп в N потоков (только directory-формат)

### 8. Что такое dead tuple и почему они появляются?

PostgreSQL не обновляет строки in-place, а создаёт новые версии (MVCC).
Старые версии — dead tuples. Они занимают место и замедляют запросы.
Их чистит autovacuum или VACUUM вручную.

### 9. Какие есть уровни изоляции транзакций?

- Read Uncommitted — в PostgreSQL работает как Read Committed
- Read Committed — по умолчанию, видит только закоммиченные данные
- Repeatable Read — снимок на момент первого запроса в транзакции
- Serializable — полная изоляция, возможны ошибки сериализации

### 10. Что такое pg_stat_statements?

Расширение, которое собирает статистику по всем выполняемым запросам:
количество вызовов, общее время, среднее время, чтение/запись блоков.
Основной инструмент для поиска медленных запросов в production.
